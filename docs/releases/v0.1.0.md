# ThreatGuard v0.1.0 — Community Preview

The first public release of ThreatGuard: automated, AI-assisted threat modeling
that turns a description of a system into a structured, methodology-backed threat
model — with consistent scoring, a data-flow diagram, and exportable reports.

Self-hosted, rules-first, and works fully offline. AI is optional enrichment, not
a dependency. **Treat outputs as a draft for human review, not a sign-off.**

## ✨ New features

- **Five methodologies** applied by a deterministic rule engine: STRIDE, DREAD,
  LINDDUN, PASTA, and OWASP Top 10 — no API key required.
- **Rich scoring & mapping:** CVSS 3.1 & 4.0, CWE, MITRE ATT&CK technique/tactic,
  and SOC 2 / ISO 27001 / PCI-DSS control mapping.
- **Trust boundaries & data-flow diagram:** boundaries are inferred automatically
  when undefined, cross-boundary flows are flagged, and a labelled DFD is
  rendered (with an interactive editor).
- **Diagram upload:** drop in a PNG/JPEG/WebP architecture diagram; a vision model
  turns it into a system model (with an editable fallback when no model is set).
- **Multi-LLM, or none:** Anthropic (Claude) or any OpenAI-compatible endpoint
  (OpenAI, Azure, Ollama, vLLM, …), auto-detected from your key — or run fully
  offline in rules-only mode.
- **Team workflow:** Release → Feature → Threat Model hierarchy, role-based access
  (user / management / admin), per-threat status tracking, release-to-release
  diffs, read-only share links, custom rules, and an audit log.
- **Reports:** HTML (interactive, with the DFD), PDF, Markdown, a CSV risk
  register, and an executive summary.
- **One-command setup** (`make dev` / `run.sh`) and a ready-to-use `examples/`
  set (system definitions + generated reports).

## 🔧 Improvements

- Consolidated, drift-free documentation with a single canonical README.
- Unified internal version reporting.
- A CLI (`atm_cli.py`) that analyzes a system file and can gate a CI pipeline via
  its exit code.
- Repository hygiene: removed committed build artifacts and broken Docker files;
  organized examples and audit records.

## 🔒 Security

- JWT auth with **refresh-token rotation** and logout revocation; bcrypt hashing;
  account lockout; per-IP rate limiting.
- **Secure by default** — every data endpoint requires a session, verified by an
  automated sweep that calls every route anonymously; RBAC + per-resource
  ownership (IDOR-tested).
- Fixed a **stored XSS** in the HTML report and hardened **CORS** (no wildcard
  with credentials).
- Patched dependency CVEs (`jinja2`, `python-multipart`); `pip-audit` clean.
- CI (lint + tests on Python 3.11 & 3.12), **CodeQL**, **secret scanning**, and
  **Dependabot** enabled; a control-by-control `SECURITY_ARCHITECTURE.md`.

## ⚠️ Known limitations

- **AI output is experimental** and quality depends on the model you choose; the
  rule engine is the authoritative, reproducible source.
- **Human review required** — not an audit, pentest, or compliance certification.
- **Single-instance / SQLite**; not designed to isolate mutually-distrusting
  tenants; rate-limit and lockout state are in-process.
- Trust-boundary inference is a heuristic; deeply event-driven topologies are only
  partially modeled.
- No Content-Security-Policy yet; CSV exports aren't formula-injection hardened.

See [`KNOWN_LIMITATIONS.md`](../../KNOWN_LIMITATIONS.md) for the full account.
