# Pentest Report Generator

Generate pentest deliverable reports as DOCX from a single YAML file.

Drop your findings into a YAML file, run one command, get a navy-themed,
ready-to-send Word document. Supports three templates: `web`, `mobile`, `api`.

---

## Quickstart

```bash
# 1. Sanity-check the install
node generate-report.js --help

# 2. Generate a report from an example
node generate-report.js examples/web-example.yaml

# That's it — a .docx is written next to the YAML.
```

You can also point it at a custom output:

```bash
node generate-report.js examples/web-example.yaml --out Q1-WebReport.docx
```

To check that your YAML parses and passes the schema without generating
a file:

```bash
node generate-report.js examples/web-example.yaml --validate-only
```

## Requirements

- **Node.js 18+** (any modern Node).
- **`docx`** npm package (already on this system — globally installed at
  `/home/claude/.npm-global/lib/node_modules`).

If you copy this CLI to another machine, install `docx` globally:

```bash
npm install -g docx@8
```

Or locally inside the CLI directory:

```bash
cd report-cli/
npm install docx@8
```

No other dependencies. The YAML parser is bundled — no `js-yaml` install
needed.

## Typical workflow

1. **Copy an example.** Pick `web-example.yaml`, `mobile-example.yaml`, or
   `api-example.yaml` and copy it to a new file:

   ```bash
   cp examples/web-example.yaml engagements/ENG-2025-Q1-WEB-OB.yaml
   ```

2. **Edit.** Update the `engagement:` block (title, dates, scope) and add
   one `findings:` entry per real finding.

3. **Validate often.** While editing:

   ```bash
   node generate-report.js engagements/ENG-2025-Q1-WEB-OB.yaml --validate-only
   ```

   The validator prints every issue at once — fix and rerun.

4. **Generate.** When clean:

   ```bash
   node generate-report.js engagements/ENG-2025-Q1-WEB-OB.yaml
   ```

5. **Open the DOCX.** Open in Word, make any visual tweaks (add screenshots,
   tweak phrasing), save. The structure is preserved.

## YAML schema reference

### Top-level

```yaml
engagement: { ... }    # required — engagement metadata
findings:   [ ... ]    # required — list of findings (at least 1)
```

### `engagement` block

| Field             | Required | Notes                                                            |
|-------------------|----------|------------------------------------------------------------------|
| `template`        | optional | `web` (default), `mobile`, or `api` — picks the methodology text |
| `title`           | yes      | Product/asset name. Shown on cover and in header.                |
| `subtitle`        | yes      | Report subtitle (e.g. "Web Application Penetration Test Report") |
| `client`          | yes      | Recipient team / organisation                                    |
| `engagement_id`   | yes      | Engagement identifier (e.g. `ENG-2025-Q1-WEB-OB`)                |
| `period`          | yes      | Testing window (e.g. `13 Jan 2025 — 31 Jan 2025`)                |
| `classification`  | yes      | Document classification line                                     |
| `version`         | yes      | Report version (string, e.g. `"1.0"`)                            |
| `author`          | yes      | Authoring team                                                   |
| `summary`         | optional | List of exec-summary paragraphs                                  |
| `summary_close`   | optional | Closing paragraph after the severity stats                       |
| `scope_in`        | optional | List of `[asset, type, notes]` triples                           |
| `scope_out`       | optional | List of strings                                                  |
| `methodology_extra` | optional | Extra paragraph after the template-default methodology         |
| `recommendations` | optional | List of strings OR `{title, body}` objects                       |
| `retest_plan`     | optional | Free text                                                        |
| `tooling`         | optional | List of strings                                                  |
| `distribution`    | optional | Distribution list for the Document Control section              |
| `retention`       | optional | Retention policy for the Document Control section              |

### `findings` list — each finding

| Field              | Required | Notes                                                               |
|--------------------|----------|---------------------------------------------------------------------|
| `id`               | yes      | Unique within the file (e.g. `WEB-001`)                             |
| `title`            | yes      | Short title                                                         |
| `severity`         | yes      | One of: `Critical`, `High`, `Medium`, `Low`, `Info`                 |
| `status`           | yes      | One of: `Open`, `In Progress`, `Remediated`, `Accepted`             |
| `component`        | yes      | The specific endpoint / class / path                                |
| `description`      | yes      | What the vulnerability is                                           |
| `impact`           | yes      | Business impact of exploitation                                     |
| `cvss`             | optional | CVSS string                                                         |
| `owasp`            | optional | OWASP category reference (e.g. `A01:2021`, `M03:2024`, `API1:2023`) |
| `cwe`              | optional | CWE reference (e.g. `CWE-639`)                                      |
| `affected_assets`  | optional | List of strings                                                     |
| `repro`            | optional | List of reproduction step strings (numbered automatically)          |
| `evidence_request` | optional | Code block — request capture                                        |
| `evidence_response`| optional | Code block — response capture                                       |
| `evidence_note`    | optional | Free-text caption for the evidence                                  |
| `remediation`      | optional | List of remediation guidance strings (bulleted)                     |
| `references`       | optional | List of reference strings (bulleted)                                |

### YAML conventions

- **Multi-line strings** — use `|` for verbatim blocks (code, JSON):

  ```yaml
  evidence_request: |
    POST /api/transfer HTTP/1.1
    Host: api.bank.example
    Authorization: Bearer eyJ...

    { "amount": 100 }
  ```

- **Lists of strings** — one item per line:

  ```yaml
  remediation:
    - Add a server-side ownership check.
    - Move to opaque ULIDs.
  ```

- **Avoid colons in unquoted values.** YAML treats `:` specially.
  Either escape or wrap in quotes:

  ```yaml
  cvss: "9.1 (CVSS:3.1/AV:N/AC:L/...)"
  ```

- **Recommendations can be plain strings or objects** with `title` + `body`:

  ```yaml
  recommendations:
    - Add SBOM scanning to CI.
    - title: Centralise authorization
      body: Implement a single authorization middleware applied to every route.
  ```

## Templates

The `template` field in `engagement:` controls three things:

- **Page header text** (`WEB PENTEST REPORT — <title>`, etc.)
- **Methodology section text** (references the matching internal methodology
  document — SEC-PT-WEB-001 / SEC-PT-MOB-001 / SEC-PT-API-001)
- **Default phrasing** in the executive summary

Everything else is identical across templates. Switch with one line.

## Running multiple reports at once

Bash loop:

```bash
for f in engagements/*.yaml; do
  node generate-report.js "$f"
done
```

Or in CI:

```yaml
# .github/workflows/reports.yml (sketch)
- run: |
    for f in engagements/*.yaml; do
      node generate-report.js "$f" --validate-only
    done
- run: |
    for f in engagements/*.yaml; do
      node generate-report.js "$f"
    done
- uses: actions/upload-artifact@v4
  with:
    name: pentest-reports
    path: engagements/*.docx
```

## Editing the rendered DOCX

The DOCX produced by the CLI is fully editable in Word/LibreOffice:

- Add screenshots — drag-and-drop into the Evidence cells.
- Tweak wording — direct edit; styling is preserved.
- Add sections — copy a finding block and modify.

The recommended pattern is: **own the YAML, regenerate when findings
change.** Treat the DOCX as a build artifact, not source. If you make
non-trivial edits in Word, copy them back into the YAML before the next
generation.

## Troubleshooting

- **`Cannot find module 'docx'`** — install it globally: `npm install -g docx@8`.
  Or set `NODE_PATH` to point at your global node_modules:

  ```bash
  export NODE_PATH=$(npm root -g)
  node generate-report.js ...
  ```

- **YAML parse error** — most often a missing space after `:` or
  inconsistent indentation. Stick to 2 spaces per level.

- **Schema validation errors** — every error is printed with the path
  (e.g. `findings[2].severity`). Fix and re-run.

- **DOCX looks visually wrong** — confirm the input file passes
  `--validate-only` first. If it does, open the DOCX in Word (not
  LibreOffice) for the best rendering; LibreOffice handles some table
  edge cases slightly differently.

- **Long lines overflowing tables** — wrap long unbreakable strings
  in your YAML at ~80 chars. The renderer keeps line breaks from `|` blocks.

## Extending

The renderer is a few hundred lines and the helpers are well-isolated:

- `lib/render.js` — orchestration. Add or remove top-level sections here.
- `lib/report.js` — visual components (cover, severity stats, finding block).
- `lib/common.js` — theme tokens, headers, page layout.

To add a new field to findings (e.g. `cwe_link` for clickable CWE IDs):

1. Add to `lib/validate.js` schema.
2. Add to `adaptFinding()` in `lib/render.js`.
3. Reference it in `R.finding()` in `lib/report.js`.

To add a new template (e.g. `cloud`):

1. Add an entry to `TEMPLATES` in `lib/render.js`.
2. Update `lib/validate.js` to allow the new template name.
